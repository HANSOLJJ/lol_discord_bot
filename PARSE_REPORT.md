## 6. "141"의 정체 — 왜 실제(150)와 다른가

**141은 실제 판 수가 아니라 `wins.json`의 내부 카운터(`total_rounds`) 값**이다.

- ROUND 표시번호 = 봇 시작 시 `total_rounds + 1` → 판마다 +1 ([got_champe.py:1160](got_champe.py#L1160), [got_champe.py:1048](got_champe.py#L1048))
- `total_rounds`도 판마다 +1 ([got_champe.py:1017](got_champe.py#L1017)), 키가 없으면 `총승수 // 3`으로 재계산 ([got_champe.py:147](got_champe.py#L147))
- 코드상 **한 판당 승수 +3 · total_rounds +1이 세트** ([got_champe.py:1009-1017](got_champe.py#L1009-L1017)) → 정상이면 `승수합 = 3 × total_rounds`가 딱 맞아야 함.

**깨진 근거**

1. **백업 내부 불일치**: 승수합 420, 그런데 3 × 141 = 423 → **정확히 1판치(3승)가 라운드는 셌는데 승수 기록이 안 됨**. 백업 자체가 어긋나 있음.
2. **카운터가 1년간 여러 번 리셋/재구축**:
   - 2025-03~09(S01–S06): ROUND가 매 세션 **1로 리셋** → 그때마다 카운터가 0 근처에서 시작.
   - 2025-11(S07): 갑자기 **ROUND 75로 점프** → 카운터가 74로 재구축된 흔적(앞의 86판과 안 맞음).
   - S07→S10: 75→141 깔끔하게 누적(+67) → 복구본(64판 + 유실 3판)과 일치.
3. **복구 150에서 어떤 세션을 빼도 백업 141과 맞는 부분집합이 없음** (차이 +9/+10/+1/+5/+4/+1이 특정 세션에 대응 안 됨).

→ **결론: 141이 구체적으로 어떤 판들인지는 데이터에 남아있지 않음(복원 불가).** 실제 판 기록의 진실은 **150판**(전부 진짜 6명, 판당 20~30분짜리 정상 판).

## 7. 테스트 판(DEV_MODE)

- 테스트는 `DEV_MODE`로 분리되어 **별도 파일 `wins_dev.json`** 에 쌓임 (`total_rounds: 97`, 가짜 ID `100000000000000001~6`, 이름 `jaecheol232` 등).
- 우리가 복구한 150판은 **전부 진짜 6명 ID**라 테스트가 섞여 있지 않음.
- 단, 진짜 계정으로 돌린 테스트가 있었다면 150에 섞였을 수 있으나 **내용상 구분 불가**(모두 정상 3:3, 20~30분).
